# План миграции OCT B-scan приложения с dearpygui на PyQt6

## Короткий ответ
Да, **PyQt6 — хороший выбор** для вашего видения: сложные окна, управляемые состояния кнопок, табличные данные с несколькими листами, асинхронные вычисления, экспорт, сохранение проекта.

## Цели миграции
1. Сохранить текущий функционал без регресса.
2. Убрать архитектурный долг (сильная связанность UI и расчетов).
3. Ввести модульную архитектуру с четкими слоями.
4. Подготовить основу для новых функций (картирование параметров, расширенный экспорт, визуализации).

## Рекомендуемая архитектура
Используйте **Clean Architecture + MVVM (или MVP)**:

- **Domain** (чистая математика и правила):
  - ROI/границы/интенсивность/μ_t/параметры.
  - Никаких зависимостей от Qt.
- **Application (Use Cases)**:
  - Оркестрация сценариев: `LoadImages`, `RunROI`, `RunBoundaries`, `RunIntensity`, `RunParameters`, `RunMuT`, `ExportResults`.
  - Правила включения/отключения кнопок.
- **Infrastructure**:
  - Чтение/запись файлов, экспорт CSV/XLSX, сериализация проекта, работа с изображениями.
- **Presentation (PyQt6)**:
  - Виджеты, модели, сигналы/слоты, таблицы, графики.

> Ключевая идея: вычислительные модули остаются переиспользуемыми и тестируемыми отдельно от GUI.

## Базовые сущности проекта
- `Project`
- `Gallery` (имя `Gallery_{i}` по умолчанию, переименование разрешено)
- `Folder` (имя `Folder_{i}` по умолчанию, связана 1:1 с `Gallery`)
- `ImageItem` (raw image + метаданные + производные артефакты)
- `RoiResult`, `BoundariesResult`, `IntensityResult`, `ParametersResult`, `MuTResult`

## Машина состояний для кнопок
Состояния на уровне галереи:

- `EMPTY`
- `IMAGES_LOADED`
- `ROI_READY`
- `BOUNDARIES_READY`
- `INTENSITY_READY`
- `PARAMETERS_READY`
- `MUT_READY`

Правила активации:
- `ROI`: активна, если в галерее есть выбранные изображения.
- `Boundaries calculation`: активна после ROI.
- `Intensity processing`: активна после `Boundaries calculation`.
- `Parameters calculation`: активна после `Boundaries calculation`, если границ ≥ 2 и нижняя граница = граница объект/среда.
- `Mu_t processing`:
  - при 1 границе или если нижняя граница не объект/среда — обязательный ручной ввод n;
  - иначе ручной ввод **или** расчет n из `Parameters calculation`.

### UI-реализация
- Центральное окно: `QMainWindow`
- Галереи/папки: `QTreeView + QStandardItemModel`
- Список изображений: `QListView/QTableView` с превью через делегат
- Таблицы результатов: `QTabWidget` + отдельная `QTableView` на лист (sheet-like)
- Графики: `pyqtgraph` (быстро) или `matplotlib` (если нужно публикационное качество)
- Диалоги задач: `QDialog` с валидацией параметров

## Потоки и производительность
- Долгие вычисления запускать через `QThreadPool + QRunnable` или `QtConcurrent`.
- UI не блокировать.
- Все промежуточные и итоговые результаты складывать в модель проекта.
- Добавить индикатор прогресса и отмену операции.

## Экспорт
Поддержать:
1. **Raw data** (CSV/NPY/JSON)
2. **Таблицы** (XLSX c несколькими листами)
3. **Изображения** (PNG/TIFF с оверлеями)
4. **Отчёт** (опционально, HTML/PDF)

## Новые функции (рекомендуемые)
1. **Картирование средних по глубине** скользящим окном:
   - Параметры окна: ширина, шаг, диапазон.
   - Результат: профиль + 2D-карта.
2. Визуализация доверительных интервалов и медианных трендов.
3. Batch-обработка нескольких галерей.
4. Журнал действий (audit trail) для воспроизводимости.

## Ответ на вопрос «переписать полностью сейчас?»
Полный «big-bang rewrite» рискован. Лучше делать **поэтапную миграцию** с сохранением ядра расчетов и поочередной заменой UI-слоя на PyQt6.

Это даст:
- контролируемые риски;
- более быстрые промежуточные результаты;
- меньше регрессий;
- прозрачную проверяемость научных/инженерных вычислений.

## Что можно сделать следующим шагом
1. Утвердить целевую структуру папок и контракты данных (`dataclasses`/`pydantic`).
2. Выделить 3–5 ключевых use-case и интерфейсы репозиториев.
3. Сделать первый вертикальный срез: `Gallery -> ROI -> Boundaries -> Table` в PyQt6.
